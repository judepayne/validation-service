## Quick Start

### Run All Tests (Recommended)
```bash
cd jvm-service

# Start server in one terminal
clojure -M -m validation-service.core

# Run tests in another terminal
./test/babashka/run-tests.clj
```

Expected output:
```
✓ All tests passed!
Total Tests: 4
Passed: 4
Failed: 0
```

### Run Individual Tests
```bash
cd jvm-service

# Test 1: Batch inline - response mode
curl -X POST http://localhost:8080/api/v1/batch \
  -H "Content-Type: application/json" \
  --data-binary @test/requests/batch-inline.json | jq .

# Test 2: Batch inline - file output mode
curl -X POST http://localhost:8080/api/v1/batch \
  -H "Content-Type: application/json" \
  --data-binary @test/requests/batch-inline-file-output.json | jq .

# Test 3: Batch-file - response mode
curl -X POST http://localhost:8080/api/v1/batch-file \
  -H "Content-Type: application/json" \
  --data-binary @test/requests/batch-file-response.json | jq .

# Test 4: Batch-file - file output mode
curl -X POST http://localhost:8080/api/v1/batch-file \
  -H "Content-Type: application/json" \
  --data-binary @test/requests/batch-file-file-output.json | jq .
```

### View Results

## Benefits of New Organization

### 1. Clear Separation of Concerns
- **test-data/** - Input data (what we're testing with)
- **requests/** - API request examples (how we call the API)
- **results/** - Output files (what the API produces)
- **babashka/** - Automation scripts (how we run tests)

### 2. Easy to Maintain
- All test files in one place
- Clear naming conventions
- Gitignored results directory (no test artifacts in git)
- Comprehensive README documentation

### 3. Reusable Test Data
- Request files can be used with curl, Postman, or scripts
- Test data files can be modified for different scenarios
- Babashka script can be extended for more tests

### 4. CI/CD Ready
- Babashka script returns exit codes (0=pass, 1=fail)
- Can be integrated into CI pipeline
- All tests can run with single command

## Adding New Tests

### 1. Add Test Data
Create new file in `test/test-data/`:
```bash
# Example: loans-with-errors.json
cat > test/test-data/loans-with-errors.json <<EOF
[
  {
    "$schema": "file:///.../loan.schema.v1.0.0.json",
    "id": "LOAN-BAD",
    "loan_number": "LN-BAD",
    "financial": {
      "principal_amount": -50000,  // Should fail
      ...
    }
  }
]
EOF
```

### 2. Create Request File
Create new file in `test/requests/`:
```bash
cat > test/requests/batch-file-error-test.json <<EOF
{
  "file_uri": "file:///.../test/test-data/loans-with-errors.json",
  "entity_type": "loan",
  "id_field": "loan_number",
  "ruleset_name": "quick",
  "output_mode": "response"
}
EOF
```

### 3. Update Test Script
Edit `test/babashka/run-tests.clj` and add to `tests` vector:
```clojure
{:name "Batch File - Error Scenario"
 :endpoint "/api/v1/batch-file"
 :request "batch-file-error-test.json"}
```

### 4. Run Tests
```bash
./test/babashka/run-tests.clj
```

## Files Generated During Tests

The following files are created in `test/results/` when tests run:
- `batch-inline-output.json` - From batch inline file output mode
- `batch-file-output.json` - From batch-file file output mode

These files are gitignored and will be regenerated on each test run.
